Netty挖掘机（十）优雅退出实现逻辑

Netty

作为高性能的异步通讯框架，Netty自身已经封装了优雅退出的实现机制。

本篇内容参考了《Netty进阶之路-跟着案例学Netty》。

回顾一下

在Linux上重新部署程序的时候，通常我们会有两个命令杀掉进程

# 正常停止
kill pid
# 粗暴停止
kill -9 pid

我们也知道，正常情况下我们不能够使用“kill -9”的方式去停止程序，这会导致很多业务问题，例如

  • 缓存数据未持久化到磁盘，导致数据丢失；
  • 正在进行写文件操作，导致文件损坏；
  • 消息队列中有消息还未处理，导致请求丢失；
  • 事务操作一半，导致数据混乱


至于Netty为什么要优雅停止，其中涉及到什么场景，继续看下去。

为什么要优雅退出

  • 尽快释放NIO线程和句柄等资源
  • 当使用flush做批量消息发送时，需要将积压在发送队列中的待发送消息发送完成
  • 正在写或正在读的消息，需要继续处理
  • 设置在NioEventLoop线程调度器中的定时任务，需要执行或清理

涉及的主要操作和资源对象

调用接口

Netty优雅退出的总入口是EventLoopGroup，在之前的《Netty挖掘机（三）结合spring搭建netty脚手架》和《Netty挖掘机（四）结合spring boot搭建netty脚手架》里，我也用到了优雅退出。

@PreDestroy
public void stop() {
    boss.shutdownGracefully();
    worker.shutdownGracefully();

事件循环线程NioEventLoop

首先我们了解一下``io.netty.util.concurrent.SingleThreadEventExecutor中对线程状态的标识静态常量值

// 未启动
private static final int ST_NOT_STARTED = 1;
// 已启动
private static final int ST_STARTED = 2;
// 退出中
private static final int ST_SHUTTING_DOWN = 3;
// 退出
private static final int ST_SHUTDOWN = 4;
// 终止
private static final int ST_TERMINATED = 5;

在后面退出的过程中， 会一直校验线程状态是否大于等于ST_SHUTTING_DOWN

public boolean isShuttingDown() {
    return state >= ST_SHUTTING_DOWN;
}


来到io.netty.util.concurrent.SingleThreadEventExecutor#shutdownGracefully

public Future<?> shutdownGracefully(long quietPeriod, long timeout, TimeUnit unit) {
    // 此时线程状态为2，已启动
    if (isShuttingDown()) {
        return terminationFuture();
    }
    // 当前线程是否事件循环原生线程
    boolean inEventLoop = inEventLoop();
    boolean wakeup;
    int oldState;
    // 自旋+原子操作，以作并发保护
    for (;;) {
        if (isShuttingDown()) {
            return terminationFuture();
        }
        int newState;
        wakeup = true;
        oldState = state;
        if (inEventLoop) {
            // 如果是事件循环原生线程，则代表是由NioEventLoop线程发起的，此时将状态变为退出中
            newState = ST_SHUTTING_DOWN;
        } else {
            // 若线程状态原来是1和2，则将状态变为退出中
            switch (oldState) {
                case ST_NOT_STARTED:
                case ST_STARTED:
                    newState = ST_SHUTTING_DOWN;
                    break;
                default:
                    newState = oldState;
                    wakeup = false;
            }
        }
        // CAS操作，修改线程状态
        if (STATE_UPDATER.compareAndSet(this, oldState, newState)) {
            break;
        }
    }
    // 代码略
    return terminationFuture();
}

1. （主线程）先判断线程状态是否已退出

2. （主线程）自旋+原子操作（CAS）将线程状态改为退出中

这里为了做并发保护，Netty4 采用了避免加锁的方式：自旋 + 原子操作，而Netty5（netty-5.0.0.Alpha1） 则是采用了加锁策略，感兴趣的可以点击前面的链接看看。


从上面代码看出，Netty 使用AtomicIntegerFieldUpdater 的 compareAndSet 对线程进行修改，若修改成功则退出循环；若发现当前线程的状态已被别的线程修改过，则继续自旋，直到发现线程已经处于ST_SHUTTING_DOWN、ST_SHUTDOWN、ST_TERMINATED的状态，才退出循环。

private static final AtomicIntegerFieldUpdater<SingleThreadEventExecutor> STATE_UPDATER =
            AtomicIntegerFieldUpdater.newUpdater(SingleThreadEventExecutor.class, "state");

继续看closeAll()，来到io.netty.channel.nio.NioEventLoop#closeAll

private void closeAll() {
    // 重新获取就绪的键
    selectAgain();
    Set<SelectionKey> keys = selector.keys();
    // 将注册在Selector上的所有channel组装
    Collection<AbstractNioChannel> channels = new ArrayList<AbstractNioChannel>(keys.size());
    for (SelectionKey k: keys) {
        Object a = k.attachment();
        if (a instanceof AbstractNioChannel) {
            channels.add((AbstractNioChannel) a);
        } else {
            // 若键不属于通道，则将其取消注册
            k.cancel();
            @SuppressWarnings("unchecked")
            NioTask<SelectableChannel> task = (NioTask<SelectableChannel>) a;
            invokeChannelUnregistered(task, k, null);
        }
    }
    // 遍历曾注册在Selector上的所有channel，将其一个个关闭
    for (AbstractNioChannel ch: channels) {
        ch.unsafe().close(ch.unsafe().voidPromise());
    }
}

再看下close方法，来到io.netty.channel.AbstractChannel.AbstractUnsafe#close(io.netty.channel.ChannelPromise, java.lang.Throwable, java.nio.channels.ClosedChannelException, boolean)

private void close(final ChannelPromise promise, final Throwable cause,
                   final ClosedChannelException closeCause, final boolean notify) {
    // 如果线程已经被取消，则直接返回
    if (!promise.setUncancellable()) {
        return;
    }
    // 是否已经发起关闭
    if (closeInitiated) {
        if (closeFuture.isDone()) {
            // Closed already.
            safeSetSuccess(promise);
        } else if (!(promise instanceof VoidChannelPromise)) { // Only needed if no VoidChannelPromise.
            // This means close() was called before so we just register a listener and return
            closeFuture.addListener(new ChannelFutureListener() {
                @Override
                public void operationComplete(ChannelFuture future) throws Exception {
                    promise.setSuccess();
                }
            });
        }
        return;
    }

    closeInitiated = true;
    // 校验通道是否处于活动状态并已连接
    final boolean wasActive = isActive();
    // 将发送队列清空，并不允许发送新的消息
    final ChannelOutboundBuffer outboundBuffer = this.outboundBuffer;
    this.outboundBuffer = null;

    // 获取准备关闭Channel的操作并返回一个执行器
    Executor closeExecutor = prepareToClose();
    if (closeExecutor != null) {
        closeExecutor.execute(new Runnable() {
            @Override
            public void run() {
                try {
                    // 执行关闭
                    doClose0(promise);
                } finally {
                    // Call invokeLater so closeAndDeregister is executed in the EventLoop again!
                    invokeLater(new Runnable() {
                        @Override
                        public void run() {
                            if (outboundBuffer != null) {
                                // Fail all the queued messages
                                outboundBuffer.failFlushed(cause, notify);
                                outboundBuffer.close(closeCause);
                            }
                            fireChannelInactiveAndDeregister(wasActive);
                        }
                    });
                }
            }
        });
    } else 
        try {
            // 在所有情况下，关闭链路并使排队的消息失败。
            doClose0(promise);
        } finally {
            if (outboundBuffer != null) {
                // 使所有排队的消息失败
                outboundBuffer.failFlushed(cause, notify);
                outboundBuffer.close(closeCause);
            }
        }
        // 当前链路是否有消息正在发送，如果有则将SelectionKey的去注册操作封装成task放到EventLoop中稍后执行
        if (inFlush0) {
            invokeLater(new Runnable() {
                @Override
                public void run() {
                    fireChannelInactiveAndDeregister(wasActive);
                }
            });
        } else {
            // 立即执行SelectionKey的去注册操作
            fireChannelInactiveAndDeregister(wasActive);
        }
    }
}

主要操作如下：

3.1 将发送队列清空，并不允许发送新的消息

if (inFlush0) {
    invokeLater(new Runnable() {
        @Override
        public void run() {
            fireChannelInactiveAndDeregister(wasActive);
        }
    });
} else {
    fireChannelInactiveAndDeregister(wasActive);
}

任务队列TaskQueue

Netty在优雅退出中，执行以上操作后，还不能够真正地退出，它仍有一些善后工作需要处理。我们知道，NioEventLoop除了进行I/O读写，还负责定时任务执行，ShutdownHook（非JDK原生）执行等。若此时有到期的定时任务，即使Channel链路关闭后仍要继续执行，线程无法退出。

1. 确认是否真正退出

在Netty执行了closeAll()关闭了链路、清空发送队列、使排队消息直接失效、SelectionKey的去注册操作 等后，还需要再次校验是否可以真正退出
io.netty.channel.nio.NioEventLoop#run

if (isShuttingDown()) {
    closeAll();
    if (confirmShutdown()) {
        return;
    }
}

来到io.netty.util.concurrent.SingleThreadEventExecutor#confirmShutdown

protected boolean confirmShutdown() {
    // 如果当前线程状态不是ST_SHUTTING_DOWN、ST_SHUTDOWN、或ST_TERMINATED，则返回false，认为还未更新线程状态
    if (!isShuttingDown()) {
        return false;
    }
    // 当前线程不是事件循环原生线程，则抛出异常，必须在事件循环线程中调用
    if (!inEventLoop()) {
        throw new IllegalStateException("must be invoked from an event loop");
    }
    // 取消所有计划任务
    cancelScheduledTasks();
    // 代码略
}

2. 执行正在TaskQueue中排队的task

来到io.netty.util.concurrent.SingleThreadEventExecutor#runAllTasks()

protected boolean runAllTasks() {
    assert inEventLoop();
    boolean fetchedAll;
    boolean ranAtLeastOne = false;

    do {
        fetchedAll = fetchFromScheduledTaskQueue();
        if (runAllTasksFrom(taskQueue)) {
            ranAtLeastOne = true;
        }
    } while (!fetchedAll); // keep on processing until we fetched all scheduled tasks.

    if (ranAtLeastOne) {
        lastExecutionTime = ScheduledFutureTask.nanoTime();
    }
    afterRunningAllTasks();
    // 若至少有一个任务执行成功，返回true
    return ranAtLeastOne;
}

3. 执行注册到NioEventLoop中的ShutdownHook

来到io.netty.util.concurrent.SingleThreadEventExecutor#runShutdownHooks

private boolean runShutdownHooks() {
    boolean ran = false;
    while (!shutdownHooks.isEmpty()) {
        List<Runnable> copy = new ArrayList<Runnable>(shutdownHooks);
        shutdownHooks.clear();
        for (Runnable task: copy) {
            try {
                task.run();
            } catch (Throwable t) {
                logger.warn("Shutdown hook raised an exception.", t);
            } finally {
                ran = true;
            }
        }
    }

    if (ran) {
        lastExecutionTime = ScheduledFutureTask.nanoTime();
    }

    return ran;
}

4. 检查是否达到优雅退出的超时时间

io.netty.util.concurrent.SingleThreadEventExecutor#confirmShutdown

// 若线程状态为ST_SHUTDOWN或ST_TERMINATED，并且达到超时时间，则确认已经退出
if (isShutdown() || nanoTime - gracefulShutdownStartTime > gracefulShutdownTimeout) {
    return true;
}
// 还未达到指定超时时间，则每隔100ms检测是否有新任务加入，有则继续执行
if (nanoTime - lastExecutionTime <= gracefulShutdownQuietPeriod) {
    // Check if any tasks were added to the queue every 100ms.
    // TODO: Change the behavior of takeTask() so that it returns on timeout.
    taskQueue.offer(WAKEUP_TASK);
    try {
        Thread.sleep(100);
    } catch (InterruptedException e) {
        // Ignore
    }

    return false;
}

挺赞的，正常的处理逻辑当退出前所有该做的操作都处理完了后，一般直接退出进程了。然而Netty还做了优化，在没达到超时时间时，继续等待新任务并执行，以确保“万无一失”。

5. Netty正式退出

当confirmShutdown()返回true后，意味着Netty优雅退出操作完成，正式退出，于是return退出了自旋的循环。NioEventLoop线程也结束，整个程序正式退出！

if (confirmShutdown()) {
    return;
}

Netty优雅退出需要注意的几点

1. 待发送的消息

调用优雅退出的方法shutdownGracefully()后，不会立即关闭链路（是在事件循环线程中执行的），ChannelOutboundBuffer中的消息可以继续发送。而本次发送操作执行完后，无论消息有没有发送出去，下一次Selector轮询中，链路都将被关闭，未发送完成的消息也会被释放和丢弃

2. 需要发送的新消息

即便ShutdownHook 触发了优雅退出方法，在退出期间应用线程仍可以随时调用Channel的write系列接口发送消息，但是这些新消息将发送失败

3. 需要执行的定时任务

诸如TaskQueue上的定时任务、普通的应用线程的定时任务、ShutdownHook，在退出时也无法保证被完全执行，完全取决于超时时间和任务数量、以及处理速度

4. 使用方式

通常用于应用进程退出时，在应用的ShutdownHook 中调用优雅退出的方法shutdownGracefully(long quietPeriod, long timeout, TimeUnit unit)接口，指定超时时间，以防止因为一些任务的执行被阻塞而无法正常退出
