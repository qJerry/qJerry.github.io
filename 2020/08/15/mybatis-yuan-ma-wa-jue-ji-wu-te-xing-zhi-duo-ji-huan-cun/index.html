Mybatis源码挖掘机（五）特性之多级缓存

Mybatis
Mybatis挖掘机
发布日期:   2020-08-15
文章字数:   3.3k
阅读时长:   13 分
阅读次数:  

Mybatis内置了一级缓存和二级缓存优化功能，默认开启一级缓存优化。

一级缓存

一级缓存默认开启，它有两个选项：SESSION（仅在一个事务内生效）、STATEMENT（仅在一个statement内生效，相当于没有一级缓存），默认是SESSION级别，作用区间SESSION大于STATEMENT。若要更改级别，可以做如下配置

配置

  • Spring

    <setting name="localCacheScope" value="SESSION"/>

  • Springboot

    mybatis-plus:
configuration:
  local-cache-scope: STATEMENT

假设一级缓存设置为SESSION级别，在一次事务中，若用户多次查询条件完全相同的SQL，Mybatis在第二次查询时，会优先命中一级缓存，避免直接对数据库重复查询，以提高性能。

原理

以上时序图逻辑在前面已经讲过了，这里只讲关于一级缓存的部分。
可以看到，关于缓存的部分，都集中在Executor内完成的，先看下关于缓存的方法

// 创建缓存key值
CacheKey createCacheKey(MappedStatement ms, Object parameterObject, RowBounds rowBounds, BoundSql boundSql);
// key值是否存在缓存
boolean isCached(MappedStatement ms, CacheKey key);
// 清除本地缓存
void clearLocalCache();

在一级缓存的操作中，主要涉及BaseExecutor和Cache两个类，

##
BaseExecutor是Executor的抽象类，提供了一些基础实现和抽象类，具体的操作一般委派给子类执行

在执行阶段，所有子类的构造函数都会默认调用BaseExecutor的构造函数，此时默认创建的缓存实现是PerpetualCache

protected BaseExecutor(Configuration configuration, Transaction transaction) {
    this.transaction = transaction;
    this.deferredLoads = new ConcurrentLinkedQueue<>();
    this.localCache = new PerpetualCache("LocalCache");
    this.localOutputParameterCache = new PerpetualCache("LocalOutputParameterCache");
    this.closed = false;
    this.configuration = configuration;
    this.wrapper = this;
}

回看时序图，Sqlsession 将查询操作委派给了执行器Executor，如果只开启了一级缓存的话，会直接执行BaseExecutor#query()

public <E> List<E> query(MappedStatement ms, Object parameter, RowBounds rowBounds, ResultHandler resultHandler) throws SQLException {
    BoundSql boundSql = ms.getBoundSql(parameter);
    CacheKey key = createCacheKey(ms, parameter, rowBounds, boundSql);
    return query(ms, parameter, rowBounds, resultHandler, key, boundSql);
}

生成key

createCacheKey适用于一级和二级缓存，了解一下生成key 的逻辑，最终的cachekey主要由mapper方法全限定名、sql分页、sql语句、方法入参、环境id组成，每次调用update()都会将属性值放入成员变量updateList，用于后续校验key值
，

public CacheKey createCacheKey(MappedStatement ms, Object parameterObject, RowBounds rowBounds, BoundSql boundSql) {
    if (closed) {
      throw new ExecutorException("Executor was closed.");
    }
    CacheKey cacheKey = new CacheKey();
    // 根据mapper方法的全限定名更新cachekey
    cacheKey.update(ms.getId());
    // 根据sql分页信息更新cachekey
    cacheKey.update(rowBounds.getOffset());
    cacheKey.update(rowBounds.getLimit());
    // 根据sql语句更新cachekey
    cacheKey.update(boundSql.getSql());
    // ......
    for (ParameterMapping parameterMapping : parameterMappings) {
      if (parameterMapping.getMode() != ParameterMode.OUT) {
        // ......
        // 根据参数更新cachekey
        cacheKey.update(value);
      }
    }
    if (configuration.getEnvironment() != null) {
      // 根据环境id更新cachekey
      cacheKey.update(configuration.getEnvironment().getId());
    }
    return cacheKey;
}

判断缓存是否相等

CacheKey内部重写了hashcode()、equals()、toString()、clone()，这里只看一下equals()，CacheKey作为key值，如何判断两个CacheKey相等呢？

可以看到，相等的条件是：两个CacheKey中由createCacheKey中组成的updateList()中每一个都相等的话，则认定为相等。

public boolean equals(Object object) {
    if (this == object) {
      return true;
    }
    if (!(object instanceof CacheKey)) {
      return false;
    }
    // 将对象强转为CacheKey
    final CacheKey cacheKey = (CacheKey) object;

    if (hashcode != cacheKey.hashcode) {
      return false;
    }
    if (checksum != cacheKey.checksum) {
      return false;
    }
    if (count != cacheKey.count) {
      return false;
    }

    for (int i = 0; i < updateList.size(); i++) {
      Object thisObject = updateList.get(i);
      Object thatObject = cacheKey.updateList.get(i);
      if (!ArrayUtil.equals(thisObject, thatObject)) {
        return false;
      }
    }
    return true;
}

设置缓存

先对当前key设置一个占位符，接着执行数据库查询操作，删除占位符，再接着设置执行数据到缓存，这个缓存仅在session内生效

private <E> List<E> queryFromDatabase(MappedStatement ms, Object parameter, RowBounds rowBounds, ResultHandler resultHandler, CacheKey key, BoundSql boundSql) throws SQLException {
    List<E> list;
    localCache.putObject(key, EXECUTION_PLACEHOLDER);
    try {
      list = doQuery(ms, parameter, rowBounds, resultHandler, boundSql);
    } finally {
      localCache.removeObject(key);
    }
    localCache.putObject(key, list);
    // 如果当前执行的是存储过程，则设置缓存到localOutputParameterCache
    if (ms.getStatementType() == StatementType.CALLABLE) {
      localOutputParameterCache.putObject(key, parameter);
    }
    return list;
}

清除缓存

在查询得出结果后，如果当前一级缓存级别是STATEMENT，则立马将其清理掉

public <E> List<E> query(MappedStatement ms, Object parameter, RowBounds rowBounds, ResultHandler resultHandler, CacheKey key, BoundSql boundSql) throws SQLException {
    if (configuration.getLocalCacheScope() == LocalCacheScope.STATEMENT) {
        // issue #482
        clearLocalCache();
    }
    return list;
}

否则在SqlSession.commit()前清理掉

public void commit(boolean required) throws SQLException {
    if (closed) {
      throw new ExecutorException("Cannot commit, transaction is already closed");
    }
    clearLocalCache();
    flushStatements();
    if (required) {
      transaction.commit();
    }
}

刷新缓存

在执行更新语句的时候，就会触发刷新缓存的操作

在更新前先清除缓存

public int update(MappedStatement ms, Object parameter) throws SQLException {
    ErrorContext.instance().resource(ms.getResource()).activity("executing an update").object(ms.getId());
    if (closed) {
      throw new ExecutorException("Executor was closed.");
    }
    clearLocalCache();
    return doUpdate(ms, parameter);
}

接着通过RoutingStatementHandler委派给PreparedStatementHandler进行更新操作

public int update(Statement statement) throws SQLException {
    PreparedStatement ps = (PreparedStatement) statement;
    ps.execute();
    int rows = ps.getUpdateCount();
    Object parameterObject = boundSql.getParameterObject();
    KeyGenerator keyGenerator = mappedStatement.getKeyGenerator();
    keyGenerator.processAfter(executor, mappedStatement, ps, parameterObject);
    return rows;
}

Cache的实现

一级缓存cache的实现是org.apache.ibatis.cache.impl.PerpetualCache，它本质上就是一个HashMap。

二级缓存

二级缓存默认关闭，若开启了默认先走二级缓存，再走一级缓存

一级缓存的缓存范围在SESSION级别，而二级缓存的缓存范围在于一个namespace。

配置

  • Spring

    <setting name="cacheEnabled" value="true"/>

  • Springboot

    mybatis-plus:
configuration:
  cache-enabled: true

在需要开启二级缓存的mapper.xml上，添加缓存节点配置

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN" "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.github.qjerry.mapper.UserMapper">
    <cache />
</mapper>

了解下cache节点的属性，参考官网官网

  • type：缓存类型，mybatis中默认是org.apache.ibatis.cache.impl.PerpetualCache

  • eviction：逐出策略，默认LRU

    • LRU，最近最少使用，删除最长时间未使用的对象
    • FIFO，先进先出，按照对象进入缓存的顺序来删除对象
    • SOFT，软引用，移除基于垃圾回收器状态和软引用规则删除对象
    • WEAK，弱引用，根据垃圾回收器状态和弱引用规则更积极地删除对象

  • flushInterval：自动刷新缓存时间间隔，一般结合ScheduledCache使用，默认为null

  • size：最多存放的引用对象个数，对于LRUCache而言，默认是1024

  • readOnly：是否可读写，默认为false，即可读写。

    • 若设置为false（可读写），则在写入时进行序列化操作，在读取时进行反序列化操作，保证在读取时对缓存对象（可理解为副本）的修改，不会影响到缓存中的对象
    • 若设置为true（仅可读），则不会有序列化操作，读取的是缓存的引用，此时对缓存对象引用的修改，会直接影响到其他地方读取缓存的操作，并发下不安全
      故默认为可读写

  • blocking：缓存命中释放锁，缓存未命中一直阻塞直到数据库获得结果，一般结合BlockingCache使用，默认为false

默认空配置的话，则代表

  • 所有select语句被缓存，所有update语句被刷新
  • 逐出策略采用LRU算法
  • 不会自动刷新缓存
  • 仅存放最多1024个引用对象
  • 支持读写缓存
  • 默认不阻塞

这里有两点要注意

  • 以上配置都完成后，Mybatis-plus封装的查询方法，默认不会走二级缓存，需要额外配置
  • 以上配置都完成后，若在同一事务内，不会走二级缓存，（开启了一级缓存则会走一级缓存）

时序图

先查询二级缓存，再查询一级缓存，否则查询数据库

原理

在二级缓存的操作中，主要涉及CachingExecutor、BaseExecutor和Cache三个类，一级缓存中，SqlSession都是直接调用BaseExecutor来做查询及缓存基础实现的，而在二级缓存中，BaseExecutor委派给了CachingExecutor，做二级缓存的实现

先看一下CachingExecutor主要的query()，先获取二级缓存的实现类，若不为空的话判断是否要刷新缓存，是否要使用二级缓存，再去实际调用底层的查询方法

public <E> List<E> query(MappedStatement ms, Object parameterObject, RowBounds rowBounds, ResultHandler resultHandler, CacheKey key, BoundSql boundSql)
      throws SQLException {
    Cache cache = ms.getCache();
    if (cache != null) {
      flushCacheIfRequired(ms);
      if (ms.isUseCache() && resultHandler == null) {
        ensureNoOutParams(ms, boundSql);
        @SuppressWarnings("unchecked")
        List<E> list = (List<E>) tcm.getObject(cache, key);
        if (list == null) {
          list = delegate.query(ms, parameterObject, rowBounds, resultHandler, key, boundSql);
          tcm.putObject(cache, key, list); // issue #578 and #116
        }
        return list;
      }
    }
    return delegate.query(ms, parameterObject, rowBounds, resultHandler, key, boundSql);
}

这里可以看到，进入query()，入参有一个MappedStatement对象，它对应着一个mapper.xml，即一个命名空间namespace，故我们可以得出一个结论，二级缓存只适用于单个命名空间。

生成key

和一级缓存的生成规则一样。

强制刷新缓存、二次确认是否要使用二级缓存

若在<select>节点中配置了flushCache为true，则代表要强制刷新缓存，适用于更新操作后缓存中的旧数据不兼容导致脏数据的场景

private void flushCacheIfRequired(MappedStatement ms) {
    Cache cache = ms.getCache();
    if (cache != null && ms.isFlushCacheRequired()) {
      tcm.clear(cache);
    }
}

若在<select>节点中配置了useCache为false，则不使用二级缓存，直接操作数据库

if (ms.isUseCache() && resultHandler == null) {}

委派模式设置待定缓存

默认情况下，开启了二级缓存后，在MappedStatement.Builder#cache打下断点，可以看到，此时缓存的组成如下图所示，利用了装饰器+委派模式，层层委派到最后一层即HashMap，来操作和获取缓存，注意，此时的缓存设置为待定缓存放在TransactionalCache.entriesToAddOnCommit中，实际设置是在commit()的时候

public void putObject(Object key, Object object) {
    entriesToAddOnCommit.put(key, object);
}

其中每个缓存实现都有对应的功能

  • SynchronizedCache：同步缓存，在每个缓存操作方法上都上了synchronized锁
  • LoggingCache：记录日志，即开启了debug级别时，如果命中了缓存则将命中率输出
  • SerializedCache：序列化缓存，在缓存写入进行serialize，在缓存读取进行deserialize
  • LruCache：lru算法，内部通过Map实现的keyMap，在每次设置缓存时用来记录缓存key值，如果此时存在需要丢弃的缓存标志eldestKey，则进行丢弃操作
  • PerpetualCache：最基础的缓存，通过HashMap实现

最完整的缓存委派链为：BlockingCache——>SynchronizedCache——>LoggineCache——>SerializedCache——>ScheduledCache——>LruCache/FifoCache/SoftCache/WeakCache——>PerpetualCache

提交并设置缓存

public void commit() {
    if (clearOnCommit) {
      delegate.clear();
    }
    flushPendingEntries();
    reset();
}

在SqlSession提交的过程中，会调用flushPendingEntries()来刷新pending中的缓存对象，即存储在TransactionalCache.entriesToAddOnCommit中的key-value，接着将缓存对象一一通过委派模式，最终存储到最底层的缓存实现PerpetualCache.cache中，下一次获取缓存也是通过委派模式从PerpetualCache.cache中获取。

时序图如下

回顾前文提到的“若在同一事务内，不会走二级缓存”，这里就给出答案了！
由于同一事务内，SqlSession不会在执行第一次查询操作后马上commit()，而上面提到了真正意义上的二级缓存设置是在commit()操作中，故无法实现二级缓存！

清除缓存

和一级缓存的清除规则一样。

刷新缓存

public int update(MappedStatement ms, Object parameterObject) throws SQLException {
    flushCacheIfRequired(ms);
    return delegate.update(ms, parameterObject);
}

在执行更新语句的时候，默认会调用flushCacheIfRequired()删除二级缓存，除非在<insert>等节点中配置了flushCache为false，则不会删除二级缓存。

private void flushCacheIfRequired(MappedStatement ms) {
    Cache cache = ms.getCache();
    if (cache != null && ms.isFlushCacheRequired()) {
      tcm.clear(cache);
    }
}

Mybatis-plus二级缓存

Mybatis-plus提供了多种内置方法来操作数据库，但是以上的二级环境配置方式对于内置方法是无法起作用的，它提供了另一种处理：添加注解

  • 添加配置
    mybatis-plus:
configuration:
  cache-enabled: true
  • 在mapper类上添加@CacheNamespace注解
    @CacheNamespace(implementation = PerpetualCache.class, eviction = LruCache.class)
public interface UserMapper extends BaseMapper<User> {
}
  • 在mapper.xml上添加节点，命名空间为以上添加了@CacheNamespace注解的mapper类
    <cache-ref namespace="com.github.qjerry.mapper.UserMapper"/>

二级缓存优化

由于一级和二级缓存都作用于本地，这对于分布式服务架构来说，在某些场景下无疑是不适用的。比如服务A查询了一次数据，服务B开始修改同一份数据后，服务A再次以相同条件查询相同数据，此时会读缓存而不会读到最新数据。故实际使用时需要使用一些中间件如redis，来实现自定义缓存对象。

同时也可以基于Spring提供的缓存机制来实现，之前写过几篇《Spring缓存挖掘机》系列，就是基于Spring的缓存机制，实现了本地 + redis分布式缓存，具体可以看下。

总结

  • 一级缓存中，默认开启SESSION级别
  • 二级缓存中，Mybatis需要在mapper.xml中配置<cache />，而对于Mybatis-plus的内置方法默认不支持二级缓存，需要显示开启@CacheNamespace注解，和mapper.xml中通过<cache-ref>来关联注解mapper类
  • 二级缓存中，多个namespace中，需要通过<cache-ref>来关联，才能共享二级缓存
  • 一级缓存作用在事务范围内，二级缓存作用在单个命名空间范围内
  • 单服务架构中，建议开启一级缓存，提升性能
  • 微服务架构中，不建议开启一级缓存，会造成脏数据，对于二级缓存，可以改成redis来重写缓存实现，否则也不建议开启
