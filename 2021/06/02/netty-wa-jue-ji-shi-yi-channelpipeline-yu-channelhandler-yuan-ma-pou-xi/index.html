Netty挖掘机（十一）ChannelPipeline与ChannelHandler源码剖析

Netty

文章字数:   2.6k

前面几篇EventLoop的底层原理，了解了事件循环执行器的创建和执行过程，接下来我们开始来深入了解Netty的数据管道ChannelPipeline和I/O事件处理器ChannelHandler，以及它们的工作原理。

准备

在分析之前，我们要先明确两点：

  1. Netty中涉及大量的inEventLoop()的判断，即判断当前是否是I/O线程，是则直接操作，否则需要异步操作；
  2. Netty中提供了事件传播机制，分别是入站事件、出站事件，入站是往链表后继节点的方向传播，出站则相反。

如下图所示，ChannelHandler的工作机制类似于Servlet中的Filter机制，采用一种类似职责链模式的方式，数据管道ChannelPipeline作为编排管理容器，内部持有一组I/O处理器ChannelHandler双向链表，I/O处理器通过每个处理器上下文ChannelHandlerContext和管道交互。消息在管道中传递，I/O处理器列表会按序逐个拦截并处理消息，故我们可以很灵活地穿插一个新的拦截处理器到管道中且不会影响其他的。

什么是Channel

Channel是Netty提供用于关联网络socket或者具有I/O功能的组件。它提供了诸如读就绪、写就绪、连接就绪、绑定就绪等的操作。

作为一个接口，它还继承了ChannelOutboundInvoker出站调用者接口，主要提供了如下功能

  • 通道的当前状态
// Channel是否已打开，且以后可能会激活
boolean isOpen();
// Channel是否已注册到EventLoop上
boolean isRegistered();
// Channel是否处于激活状态，且已连接
boolean isActive();
// 当且仅当I/O线程立即执行请求的写操作时，返回true
boolean isWritable();
  • 通道的配置信息
// Channel所注册的EventLoop
EventLoop eventLoop();
// Channel配置项
ChannelConfig config();
// Channel绑定的本地地址
SocketAddress localAddress();
// Channel绑定的远程地址
SocketAddress remoteAddress();
// 获取直到isWritable()返回false为止，可以写入的字节数，在返回false之前大于0，返回false则为0
long bytesBeforeUnwritable();
// 获取必须从底层缓冲区中耗尽多少字节数，直到isWritable()返回true为止
long bytesBeforeWritable();
// 返回分配到的ChannelPipeline
ChannelPipeline pipeline();
// 返回分配到的ByteBufAllocator，用于分配ByteBuf
ByteBufAllocator alloc();
  • 通道支持的I/O操作
// 绑定操作。绑定到指定的localAddress，并在操作完成后通知
ChannelFuture bind(SocketAddress localAddress);
// 连接操作。连接到指定的remoteAddress，并在操作完成后通知
ChannelFuture connect(SocketAddress remoteAddress);
// 取消连接操作。请求与远程方断开连接，并在操作完成后通知
ChannelFuture disconnect();
// 关闭Channel操作
ChannelFuture close();
// 读取操作。请求从Channel读取数据到第一个入站缓冲区，并触发ChannelInboundHandler#channelRead，当数据已读取时则触发ChannelInboundHandler#channelReadComplete
ChannelOutboundInvoker read();
// 写操作。请求通过当前ChannelHandlerContext，来经过ChannelPipeline写入消息。此方法不会冲刷消息，故还需要调用flush()
ChannelFuture write(Object msg);
// 冲刷操作。请求通过当前ChannelOutboundInvoker，来冲刷所有未决消息
ChannelOutboundInvoker flush();
// 写操作和冲刷操作的结合
ChannelFuture writeAndFlush(Object msg, ChannelPromise promise);

源码中也提到了，Netty中涉及I/O操作的均是异步操作，这意味着任意的I/O操作都会立即返回，并且会通过执行I/O操作所返回的ChannelFuture实例获取操作结果，以此来实现。

为什么Netty是一个异步事件驱动的框架?
从I/O模型上看，它采用了NIO，即同步非阻塞的方式（当然也有其他的实现方式，如异步非阻塞）。但是何为异步事件驱动，即所有的I/O操作均是异步的，每次执行都会立即返回，且返回的一个是ChannelFuture实例，可得知这一次I/O操作的成功与否。

什么是ChannelHandler

贴一下源码注释：
Handles an I/O event or intercepts an I/O operation, and forwards it to its next handler in its.

ChannelHandler是在Channel的生命周期一系列操作中的处理器。

让我们回顾一下，在通过Netty实现的API项目中，我们都用到了ChannelHandler来处理、转发消息

public class HttpServerHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) throws Exception {
        try {
            // do something...
        } finally {
            ReferenceCountUtil.release(msg);
        }
    }

}

而ChannelHandler是在每一次I/O事件进来时调用addLast按序添加到管道中，addLast是通过更新链表的指针位置来添加的。

ServerBootstrap bootstrap = new ServerBootstrap();
bootstrap
    // 绑定两个组
    .group(boss, worker)
    // 创建NioServerSocketChannel实例
    .channel(NioServerSocketChannel.class)
    // 添加处理器Handler
    .childHandler(new ChannelInitializer<SocketChannel>() {
        @Override
        protected void initChannel(SocketChannel channel) throws Exception {
            // 为通道Channel进行初始化配置
            ChannelPipeline pipeline = channel.pipeline();
            pipeline.addLast(new HttpRequestDecoder(),
                    new HttpResponseEncoder(),
                    new HttpServerHandler(controllerAdapter)
            );
        }
    })
bootstrap.bind(port).sync();

什么是ChannelHandlerContext

上面说了，它主要是使ChannelHandler和ChannelPipeline、其他的ChannelHandler进行交互。

这里主要讲下它的抽象实现类AbstractChannelHandlerContext，作为双向链表中的某个节点结构，它内部维护了前驱和后继节点，为ChannelHandlerContext提供了很多默认实现。其中，比较重要的知识点是管理ChannelContext的状态，用volatile关键字标识的handlerState，状态类型主要有

// 初始状态
private static final int INIT = 0;
// ChannelHandler#handlerAdded即将被调用
private static final int ADD_PENDING = 1;
// ChannelHandler#handlerAdded已经被调用完成
private static final int ADD_COMPLETE = 2;
// ChannelHandler#handlerRemoved已经被调用完成
private static final int REMOVE_COMPLETE = 3;

对于状态的切换，在后面的工作原理会提到。

接下来再分析下类图和源码

类图

先看下类图，最顶层接口ChannelHandler，其直接实现子类有出入站I/O处理器、和一个适配者处理器实现，而基于它们，Netty又分别封装了出入站的适配器处理器实现，通过适配器模式实现了更丰富的消息处理和转发的功能。

基层接口ChannelHandler

用于处理I/O事件或者拦截I/O操作，处理后转发到下一个Handler。
它提供了一个处理器上下文ChannelHandlerContext，通过上下文对象与ChannelPipeline进行交互，故它可以在上游或下游传递事件，动态地更改管道或存储信息。

// 将ChannelHandler添加到上下文中，并可以处理事件后调用
void handlerAdded(ChannelHandlerContext ctx) throws Exception;
// 将ChannelHandler从上下文中移除时，并不再处理事件后调用
void handlerRemoved(ChannelHandlerContext ctx) throws Exception;
// 触发了Throwable的处理，已被弃用，Netty建议通过实现ChannelInboundHandler接口去处理
@Deprecated
void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) throws Exception;
// 带有此注解的ChannelHandler实例被认为可共享实例，可被添加多多个数据管道`ChannelPipeline`中
// 若没指定注解，则必须创建一个新实例
@Inherited
@Documented
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@interface Sharable {
    // no value
}

入站处理器接口ChannelInboundHandler

实现了基层接口ChannelHandler，提供了入站I/O操作的各种channel状态更改后的回调方法处理。
如channel注册、取消注册、激活、停用并结束生命周期、已读、读完最后一条消息，其中还支持触发用户事件的处理、Channel写状态被改变的处理、常信息的处理。

出站处理器接口ChannelOutboundHandler

实现了基层接口ChannelHandler，提供了出站I/O操作的各种channel状态更改后的回调方法处理。
如channel绑定（到localAddress ）、连接（从localAddress 连接到remoteAddress）、取消连接、关闭、注销（从当前已注册的EventLoop）、读、写（通过ChannelPipeline刷新到Channel上，即Channel.flush()会被调用）、刷新。

入站处理器适配器类ChannelInboundHandlerAdapter

入站处理器接口ChannelInboundHandler的抽象实现类，以下实现方法仅是将操作结果转发给下一个ChannelHandler，我们可以作为子类覆盖它的实现方式，如Netty自带的SimpleChannelInboundHandler就覆盖了部分方法的实现。

在实现方法的处理上，仅仅将各个channel状态的回调进行转发到ChannelPipeline中的下一个处理器

@Skip
@Override
public void channelRegistered(ChannelHandlerContext ctx) throws Exception {
    ctx.fireChannelRegistered();
}
// ......

其中，@Skip注解的方法，被认定为ChannelPipeline直接跳过不会调用它，而只能在ChannelHandler下使用，很有意思的是，源码中还提到，区别于JDK的元注解@Inherited（使它所修饰的注解具有继承性）会将特性传递至子类，一旦子类覆盖了方法实现，则不再跳过。具体的拦截处理是在io.netty.channel.ChannelHandlerMask中进行，有兴趣的盆友可以看看。

元注解，meta-annotation，即可以注解到注解类型上的注解，JDK提供了四种元注解：@Target、@Retention、@Documented、@Inherited。

入站处理器简单抽象实现类SimpleChannelInboundHandler

继承了ChannelInboundHandlerAdapter，主要对channelRead方法的实现进行覆盖

  • 提供了泛型处理
    即允许显示地处理特定的类型的消息，比如可以指定处理类型为String的消息，或者指定FullHttpRequest的消息，一次性获取所有的请求内容，而不用涉及到对不同请求过程的处理（如HttpReuqrst、HttpContent）。
  • 自动释放消息
    对于指定类型的消息，在执行完后默认会调用ReferenceCountUtil#release释放消息内容，故当需要将消息传递给下个处理器时，应调用ReferenceCountUtil#retain

看下具体的代码

默认创建的实例，会标识为消息自动清除，如果不想自动清除的话，可以调用第二个方法

protected SimpleChannelInboundHandler() {
    this(true);
}
protected SimpleChannelInboundHandler(boolean autoRelease) {
    matcher = TypeParameterMatcher.find(this, SimpleChannelInboundHandler.class, "I");
    this.autoRelease = autoRelease;
}

这里的覆盖目的，主要在于检测消息类型是否匹配指定类型，还有支持自动释放消息，作为一个抽象类，只提供了一个抽象方法channelRead0，给子类进行消息处理。

public void channelRead(ChannelHandlerContext ctx, Object msg) throws Exception {
    boolean release = true;
    try {
        if (acceptInboundMessage(msg)) {
            @SuppressWarnings("unchecked")
            I imsg = (I) msg;
            channelRead0(ctx, imsg);
        } else {
            release = false;
            ctx.fireChannelRead(msg);
        }
    } finally {
        if (autoRelease && release) {
            ReferenceCountUtil.release(msg);
        }
    }
}

protected abstract void channelRead0(ChannelHandlerContext ctx, I msg) throws Exception;
