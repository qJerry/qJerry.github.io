上一篇主要了解了ChannelHandler相关的接口和接口方法，这一篇则着重讲解涉及到的工作原理。

工作原理

以下所讲内容，代码标准基于《Netty挖掘机（四）结合spring boot搭建netty脚手架》。

当产生一次请求时，触发读就绪事件（channelRead），I/O线程NioEventLoop开始调用ChannelPipeline的fireChannelRead()方法，将消息传递给数据管道。如本文第一幅图，其中消息会依次经过HeadContext、添加的多个ChannelHandler、TailContext，进行消息拦截和处理，这期间可以随时中断消息的传递。之后开始反方向地通过write()发送消息，将消息写入到消息缓冲区，调用flush()冲刷到远端节点。

我这里把它分为五个过程，一个是创建通道，注册通道，激活通道，执行职责链的读取消息事件、执行职责链的读取完成事件。

创建通道

在之前讲过的《Netty挖掘机 EventLoop底层原理》部分，我们了解到NioEventLoop作为I/O线程会持续监听事件，当有事件进来时会将其添加到任务队列，并异步执行。故当触发读就绪事件时，NioEventLoop会添加异步任务，注册Channel后执行管道初始化操作。

Selector轮询检测，发现有需要处理的I/O事件

代码位于NioEventLoop#processSelectedKey(SelectionKey, AbstractNioChannel)

if ((readyOps & (SelectionKey.OP_READ | SelectionKey.OP_ACCEPT)) != 0 || readyOps == 0) {
    unsafe.read();

接下来来到AbstractNioMessageChannel.NioMessageUnsafe#read，看一下read方法（忽略部分代码）

public void read() {
    // 读消息，将消息放到集合readBuf 中
    int localRead = doReadMessages(readBuf);
    if (localRead == 0) {
        break;
    }
    if (localRead < 0) {
        closed = true;
        break;
    }

    int size = readBuf.size();
    // 执行fireChannelRead传递消息
    for (int i = 0; i < size; i ++) {
        readPending = false;
        pipeline.fireChannelRead(readBuf.get(i));
    }
    readBuf.clear();
    allocHandle.readComplete();
    pipeline.fireChannelReadComplete();
}

创建通道NioSocketChannel

当触发读就绪时，调用NioUnsafe#read开始处理事件，其中调用doReadMessages时，会创建一个NioSocketChannel实例

SocketChannel ch = SocketUtils.accept(javaChannel());
if (ch != null) {
    buf.add(new NioSocketChannel(this, ch));
    return 1;
}

内部再初始化一个数据管道DefaultChannelPipeline，用于后续消息传递。前面我们提到在ChannelPipeline内部持有一组I/O处理器ChannelHandler链表，在初始化数据管道时，会先在链表头尾分别创建一个头节点处理器HeadContext和尾节点处理器TailContext。最终放到集合对象readBuf中

protected DefaultChannelPipeline(Channel channel) {
    this.channel = ObjectUtil.checkNotNull(channel, "channel");
    succeededFuture = new SucceededChannelFuture(channel, null);
    voidPromise =  new VoidChannelPromise(channel, true);

    tail = new TailContext(this);
    head = new HeadContext(this);

    head.next = tail;
    tail.prev = head;
}

传递通道对象readBuf

入口代码位于DefaultChannelPipeline#fireChannelRead。

我们知道，Netty在启动期间，会创建一个NioServerSocketChannel实例，内部也初始化了一个数据管道DefaultChannelPipeline，不同于上面的NioSocketChannel，其中的I/O处理器链表，还会穿插一个ServerBootstrapAcceptor，代码位于ServerBootstrap#init

ch.eventLoop().execute(new Runnable() {
    @Override
    public void run() {
        pipeline.addLast(new ServerBootstrapAcceptor(
                ch, currentChildGroup, currentChildHandler, currentChildOptions, currentChildAttrs));
    }
});

那么它的作用是什么？回顾下前面的启动代码，调用了ServerBootstrap#childHandler(io.netty.channel.ChannelHandler)，用于添加一个处理器ChannelInitializer，处理NioSocketChannel的请求。

ServerBootstrap bootstrap = new ServerBootstrap();
bootstrap.childHandler(new HttpChannelInitializer(dispatcherServlet));
// ......
bootstrap.bind(port).sync();

故在传递通道对象时(fireChannelRead)，会依次调用NioServerSocketChannel中数据管道所持有的链表，调用顺序为HeadContext、ServerBootstrapAcceptor、TailContext。

static void invokeChannelRead(final AbstractChannelHandlerContext next, Object msg) {
    final Object m = next.pipeline.touch(ObjectUtil.checkNotNull(msg, "msg"), next);
    EventExecutor executor = next.executor();
    // 如果当前线程是I/O线程，则执行invokeChannelRead
    if (executor.inEventLoop()) {
        next.invokeChannelRead(m);
    } else {// 否则异步执行
        executor.execute(new Runnable() {
            @Override
            public void run() {
                next.invokeChannelRead(m);
            }
        });
    }
}

这里主要看ServerBootstrapAcceptor#channelRead，内部逻辑是将启动代码中的childHandler，添加到NioSocketChannel，即将HttpChannelInitializer插入到链表中

public void channelRead(ChannelHandlerContext ctx, Object msg) {
    final Channel child = (Channel) msg;

    child.pipeline().addLast(childHandler);

    setChannelOptions(child, childOptions, logger);
    setAttributes(child, childAttrs);
}

接着将NioSocketChannel注册到EventLoop中，主要是利用执行器异步执行注册方法AbstractChannel.AbstractUnsafe#register0，并添加了监听器，监听异步操作完成的回调。

try {

注册通道

上一步将注册方法放置任务队列中执行，执行入口在之前讲过的《Netty挖掘机 EventLoop底层原理》部分也已经谈到了，NioEventLoop#run会不断轮询任务队列任务。

看下注册代码，总共有3个步骤

private void register0(ChannelPromise promise) {
    try {
        // 检查是否需要添加处理器
        pipeline.invokeHandlerAddedIfNeeded();
        safeSetSuccess(promise);
        // 若通道注册完成，则pipeline会调用fireChannelRegistered 触发注册完成事件
        pipeline.fireChannelRegistered();
        if (isActive()) {
            if (firstRegistration) {
                // 触发通道激活事件
                pipeline.fireChannelActive();
            } else if (config().isAutoRead()) {
                beginRead();
            }
        }
    } catch (Throwable t) {
        closeForcibly();
        closeFuture.setClosed();
        safeSetFailure(promise, t);
    }
}

再看下时序图，包含了3个步骤的具体流程，很清晰。很明显这一块大多都是ChannelPipeline和ChannelHandlerContext在交互。

检查是否需要添加I/O处理器

final void invokeHandlerAddedIfNeeded() {
    // 确认是否在I/O线程里，是否是第一次注册
    assert channel.eventLoop().inEventLoop();
    if (firstRegistration) {
        firstRegistration = false;
        callHandlerAddedForAllHandlers();
    }
}
加锁控制注册过程

  • 针对ChannelPipeline实例上锁，此时将registered置为true，目的在于标识pipeline关联的Channel已经被注册，一旦注册即不再会有新的处理器加入，后续操作亦会沿用此状态；

  • 将全局变量的pendingHandlerCallbackHead的引用赋给局部变量pendingHandlerCallbackHead并置为空，目的在于便于GC及时回收；

  • 上面的引用赋值操作在锁外部执行，目的在于避免死锁，当持有锁期间调用内层的handlerAdded()时，在I/O线程外部执行了handlerAdded()新增一个handler时，会产生死锁。

    private void callHandlerAddedForAllHandlers() {
  final PendingHandlerCallback pendingHandlerCallbackHead;
  synchronized (this) {
      assert !registered;
      registered = true;

      pendingHandlerCallbackHead = this.pendingHandlerCallbackHead;
      this.pendingHandlerCallbackHead = null;
  }
  PendingHandlerCallback task = pendingHandlerCallbackHead;
  while (task != null) {
      task.execute();
      task = task.next;
  }
}

了解下PendingHandlerCallback，作为DefaultChannelPipeline中的抽象静态内部类，它还是一个Runnable，内部维护了处理器上下文ctx和next节点，定义了一个抽象方法execute()，用于直接提交自身到I/O线程EventLoop中去执行。

private abstract static class PendingHandlerCallback implements Runnable {
    final AbstractChannelHandlerContext ctx;
    PendingHandlerCallback next;
    PendingHandlerCallback(AbstractChannelHandlerContext ctx) {
        this.ctx = ctx;
    }
    abstract void execute();
}

可以简单看下它的子类PendingHandlerAddedTask和PendingHandlerRemovedTask，分别用于添加handler和移除handler

private final class PendingHandlerAddedTask extends PendingHandlerCallback {
    PendingHandlerAddedTask(AbstractChannelHandlerContext ctx) {
        super(ctx);
    }

    @Override
    public void run() {
        callHandlerAdded0(ctx);
    }

    @Override
    void execute() {
        EventExecutor executor = ctx.executor();
        // 若在I/O线程中则直接调用
        if (executor.inEventLoop()) {
            callHandlerAdded0(ctx);
        } else {// 否则异步调用
            try {
                executor.execute(this);
            } catch (RejectedExecutionException e) {
                if (logger.isWarnEnabled()) {
                    logger.warn(
                            "Can't invoke handlerAdded() as the EventExecutor {} rejected it, removing handler {}.",
                            executor, ctx.name(), e);
                }
                atomicRemoveFromHandlerList(ctx);
                ctx.setRemoved();
            }
        }
    }
}
添加处理器，将ChannelContext状态置为ADD_COMPLETE

调用链路来到callHandlerAdded()，开始管理ChannelContext的状态。对于handler的状态可以往上翻”什么是ChannelHandlerContext“

final void callHandlerAdded() throws Exception {
    // We must call setAddComplete before calling handlerAdded. Otherwise if the handlerAdded method generates
    // any pipeline events ctx.handler() will miss them because the state will not allow it.
    if (setAddComplete()) {
        handler().handlerAdded(this);
    }
}

在执行handlerAdded()之前，即添加handler到链表之前，必须先将状态置为ADD_COMPLETE，这里使用CAS的方式进行修改

final boolean setAddComplete() {
    for (;;) {
        int oldState = handlerState;
        if (oldState == REMOVE_COMPLETE) {
            return false;
        }
        // Ensure we never update when the handlerState is REMOVE_COMPLETE already.
        // oldState is usually ADD_PENDING but can also be REMOVE_COMPLETE when an EventExecutor is used that is not
        // exposing ordering guarantees.
        if (HANDLER_STATE_UPDATER.compareAndSet(this, oldState, ADD_COMPLETE)) {
            return true;
        }
    }
}
基于添加的处理器，调用handlerAdded

在上面提到，此时这里添加的处理器是ChannelInitializer，即初始化通道处理器，一般会往管道中添加多个I/O事件处理器，最常用的是解码器和编码器，还有自定义的业务处理器。

如果registered为true，则开始初始化通道，且当初始化处理器执行完后，需要从链表中移除该处理器，避免多次初始化！

public void handlerAdded(ChannelHandlerContext ctx) throws Exception {
    if (ctx.channel().isRegistered()) {
        if (initChannel(ctx)) {
            removeState(ctx);
        }
    }
}
private boolean initChannel(ChannelHandlerContext ctx) throws Exception {
    if (initMap.add(ctx)) { // Guard against re-entrance.
        try {
            initChannel((C) ctx.channel());
        } catch (Throwable cause) {
            exceptionCaught(ctx, cause);
        } finally {
            ChannelPipeline pipeline = ctx.pipeline();
            if (pipeline.context(this) != null) {
                pipeline.remove(this);
            }
        }
        return true;
    }
    return false;
}

初始化的逻辑需要看对应的实现类，如以下是自定义的实现，添加多个I/O事件处理器，常用的解码器和编码器，还有自定义的业务处理器。

protected void initChannel(Channel ch) throws Exception {
    ChannelPipeline pipeline = ch.pipeline();
    pipeline.addLast(new HttpRequestDecoder());
    pipeline.addLast(new HttpResponseEncoder());
    pipeline.addLast(new HttpFirstServerHandler());
    pipeline.addLast(new HttpLastServerHandler(dispatcherServlet));
    log.debug("添加处理器到管道...");
}

经过初始化操作后，此时链表的节点顺序为HeadContext、ChannelInitializer、HttpRequestDecoder、HttpResponseEncoder、HttpFirstServerHandler、HttpLastServerHandler、TailContext。

移除处理器，将ChannelContext状态置为REMOVE_COMPLETE

上面提到初始化后，需要移除处理器，内部调用了DefaultChannelPipeline#callHandlerRemoved0。

final void callHandlerRemoved() throws Exception {
    try {
        // Only call handlerRemoved(...) if we called handlerAdded(...) before.
        if (handlerState == ADD_COMPLETE) {
            // 调用处理器中的handlerRemoved
            handler().handlerRemoved(this);
        }
    } finally {
        // Mark the handler as removed in any case.
        setRemoved();
    }
}
final void setRemoved() {
    handlerState = REMOVE_COMPLETE;
}

经过移除操作后，此时链表的节点顺序为HeadContext、HttpRequestDecoder、HttpResponseEncoder、HttpFirstServerHandler、HttpLastServerHandler、TailContext。

注册完成（事件传播channelRegistered）

上一步通道注册并初始化完成后，则开始触发注册完成事件。

这里会触发入站的注册完成事件传播，传递于channelRegistered。

  • 调用pipeline的fireChannelRegistered，开启事件传递，由于当前事件是入站事件，入站事件是往链表后继节点方向传递的，故这里首先传入的是链表的head节点，即HeadContext。
    public final ChannelPipeline fireChannelRegistered() {
  AbstractChannelHandlerContext.invokeChannelRegistered(head);
  return this;
}
  • 传入头结点后，AbstractChannelHandlerContext接着调用内部的invokeChannelRegistered方法，把事件转发给DefaultChannelPipeline.HeadContext#channelRegistered。
    public void channelRegistered(ChannelHandlerContext ctx) {
  // 二次确认是否要检测
  invokeHandlerAddedIfNeeded();
  ctx.fireChannelRegistered();
}
    看下AbstractChannelHandlerContext#fireChannelRegistered，其中调用的findContextInbound，在于查找最近的入站处理器，接着再调用invokeChannelRegistered 把事件传递传递到该处理器的channelRegistered。这个过程就是入站激活事件的传播过程。在示例代码中，这里只会传播到DefaultChannelPipeline.TailContext#channelRegistered。
    public ChannelHandlerContext fireChannelRegistered() {
  invokeChannelRegistered(findContextInbound(MASK_CHANNEL_REGISTERED));
  return this;
}

激活通道（事件传播channelActive）

激活的最终目的在于触发真正的读就绪操作。

上一步触发通道注册成功事件后，接下来将走激活通道的流程。

if (isActive()) {
    if (firstRegistration) {
        pipeline.fireChannelActive();
    } else if (config().isAutoRead()) {
        // This channel was registered before and autoRead() is set. This means we need to begin read
        // again so that we process inbound data.
        //
        // See https://github.com/netty/netty/issues/4805
        beginRead();
    }
}

这里会触发入站的激活事件传播，传递于channelActive。

  • 先判断通道是否处于激活状态并且已连接，是否首次注册，满足则开始触发通道激活事件。此时调用pipeline的fireChannelActive方法，由于当前事件是入站事件，入站事件是往链表后继节点方向传递的，故这里首先传入的是链表的head节点，即HeadContext

    public final ChannelPipeline fireChannelActive() {
  AbstractChannelHandlerContext.invokeChannelActive(head);
  return this;
}

  • 传入头结点后，AbstractChannelHandlerContext接着调用内部的invokeChannelActive方法，把事件转发给，DefaultChannelPipeline.HeadContext#channelActive

    public void channelActive(ChannelHandlerContext ctx) {
  ctx.fireChannelActive();
  // 自动读取数据
  readIfIsAutoRead();
}

    看下AbstractChannelHandlerContext#fireChannelActive，其中调用的findContextInbound，在于往后继节点方向查找最近的入站处理器，接着再调用invokeChannelActive 把事件传递传递到该处理器的channelActive。这个过程就是入站激活事件的传播过程。在示例代码中，这里只会传播到DefaultChannelPipeline.TailContext#channelActive

    public ChannelHandlerContext fireChannelActive() {
  invokeChannelActive(findContextInbound(MASK_CHANNEL_ACTIVE));
  return this;
}
自动读取数据（事件传播）

如果配置了通道自动读取数据，则需要调用NioSocketChannel#read，这一步主要是为了触发再一次的读就绪操作，开始真正的数据传输。

private void readIfIsAutoRead() {
    if (channel.config().isAutoRead()) {
        channel.read();
    }
}

里面的链路相对简单，通过调用DefaultChannelPipeline#read，从tail节点开始调用TailContext#read。查看源码可知，read方法是在ChannelOutboundInvoker接口定义的，这个接口定义的是出站事件的调用方法。故在下面的代码中，我们可以看到Outbound的字眼。
调用AbstractChannelHandlerContext#findContextOutbound，即从tail节点开始，逐步往前驱节点方向遍历最近的有出站事件执行的处理器并返回，在示例代码中，这里只会传播到DefaultChannelPipeline.HeadContext#read

public ChannelHandlerContext read() {
    final AbstractChannelHandlerContext next = findContextOutbound(MASK_READ);
    EventExecutor executor = next.executor();
    if (executor.inEventLoop()) {
        next.invokeRead();
    } else {
        Tasks tasks = next.invokeTasks;
        if (tasks == null) {
            next.invokeTasks = tasks = new Tasks(next);
        }
        executor.execute(tasks.invokeReadTask);
    }

    return this;
}

最后调用了NioSocketChannel#doBeginRead，即给当前的NioSocketChannel注册读就绪事件。

protected void doBeginRead() throws Exception {
    // Channel.read() or ChannelHandlerContext.read() was called
    final SelectionKey selectionKey = this.selectionKey;
    if (!selectionKey.isValid()) {
        return;
    }

    readPending = true;

    final int interestOps = selectionKey.interestOps();
    // 与操作为0，则将其注册
    if ((interestOps & readInterestOp) == 0) {
        // 重新设置为感兴趣的读就绪事件
        selectionKey.interestOps(interestOps | readInterestOp);
    }
}

这一步执行完后，将再次触发读就绪事件，上次的读就绪事件来源，是为了创建和注册通道，这次的读就绪事件则是为了下一步。

执行职责链的读取消息事件（入站）

先回顾下，基于示例代码，之前我们整理得到的一次请求入站的链表顺序是HeadContext、HttpRequestDecoder、HttpResponseEncoder、HttpFirstServerHandler、HttpLastServerHandler、TailContext。

监听读就绪事件后，来到AbstractNioByteChannel.NioByteUnsafe#read

do {
    byteBuf = allocHandle.allocate(allocator);
    allocHandle.lastBytesRead(doReadBytes(byteBuf));
    if (allocHandle.lastBytesRead() <= 0) {
        // nothing was read. release the buffer.
        byteBuf.release();

        close = allocHandle.lastBytesRead() < 0;
        if (close) {

} while (allocHandle.continueReading());

上一步读取消息并在链表的处理器上做了处理后，还需要将处理结果写到缓存区

在自定义的处理器HttpLastServerHandler中，调用ChannelOutboundInvoker#write(java.lang.Object)开始发送消息，同样来到AbstractChannelHandlerContext#write(java.lang.Object, boolean, io.netty.channel.ChannelPromise)。由于当前事件是出站事件，出站事件是往链表前继节点方向传递的，故调用findContextOutbound逐个获取前驱节点处理器，并选择性调用invokeWriteAndFlush或者invokeWrite将消息发送出去

private void write(Object msg, boolean flush, ChannelPromise promise) {
    ObjectUtil.checkNotNull(msg, "msg");
    try {
        if (isNotValidPromise(promise, true)) {
            ReferenceCountUtil.release(msg);
            // cancelled
            return;
        }
    } catch (RuntimeException e) {
        ReferenceCountUtil.release(msg);
        throw e;
    }

    final AbstractChannelHandlerContext next = findContextOutbound(flush ?
            (MASK_WRITE | MASK_FLUSH) : MASK_WRITE);
    final Object m = pipeline.touch(msg, next);
    EventExecutor executor = next.executor();
    if (executor.inEventLoop()) {
        if (flush) {
            next.invokeWriteAndFlush(m, promise);
        } else {
            next.invokeWrite(m, promise);
        }
    } else {
        final AbstractWriteTask task;
        if (flush) {
            task = WriteAndFlushTask.newInstance(next, m, promise);
        }  else {
            task = WriteTask.newInstance(next, m, promise);
        }
        if (!safeExecute(executor, task, promise, m)) {
            // We failed to submit the AbstractWriteTask. We need to cancel it so we decrement the pending bytes
            // and put it back in the Recycler for re-use later.
            //
            // See https://github.com/netty/netty/issues/8343.
            task.cancel();
        }
    }
}

由于出站事件涉及到编码器处理，故这里针对消息的处理是在HttpResponseEncoder中进行，此时会调用MessageToMessageEncoder#write。处理过程如下

  • 判断当前消息msg是否应处理，即消息类型是否属于HttpObject、ByteBuf、FileRegion，并且不属于HttpRequest

    • 满足则进行下一步
        public boolean acceptOutboundMessage(Object msg) throws Exception {
      return super.acceptOutboundMessage(msg) && !(msg instanceof HttpRequest);
  }
  public boolean acceptOutboundMessage(Object msg) throws Exception {
      return msg instanceof HttpObject || msg instanceof ByteBuf || msg instanceof FileRegion;
  }
    • 不满足则直接调用write方法将消息发送到缓冲区
        if (acceptOutboundMessage(msg)) {
  } else {
      ctx.write(msg, promise);
  }

  • 创建CodecOutputList实例，调用HttpObjectEncoder#encode方法将消息编码并组装成一个个ByteBuf赋给它，后再将消息对象msg进行释放

CodecOutputList out = null;
out = CodecOutputList.newInstance();
@SuppressWarnings("unchecked")
I cast = (I) msg;
try {
    encode(ctx, cast, out);
} finally {
    ReferenceCountUtil.release(cast);
}

if (out.isEmpty()) {
    out.recycle();
    out = null;

    throw new EncoderException(
            StringUtil.simpleClassName(this) + " must produce at least one message.");
}
  • 如果CodecOutputList集合不为空，并且长度为1，则直接调用write方法将消息发送到缓冲区，否则逐个遍历发送

在遍历过程中，源码提到了，检查是否可以使用voidPromise进行额外的写操作以减少GC压力。这里暂时不多讲。

if (out != null) {
    final int sizeMinusOne = out.size() - 1;
    if (sizeMinusOne == 0) {
        ctx.write(out.getUnsafe(0), promise);
    } else if (sizeMinusOne > 0) {
        if (promise == ctx.voidPromise()) {
            writeVoidPromise(ctx, out);
        } else {
            writePromiseCombiner(ctx, out, promise);
        }
    }
    out.recycle();
}

执行职责链的读取完成事件（出站）

当执行了读取消息事件pipeline.fireChannelRead后，将要触发读取完成事件，传递于channelReadComplete。

allocHandle.readComplete();
pipeline.fireChannelReadComplete();

和前面的传播类似，往链表后继节点方向逐个调用处理器的channelReadComplete，处理读取成功后的执行事项。在示例代码中，HttpLastServerHandler、HttpFirstServerHandler都实现了该方法。

public void channelReadComplete(ChannelHandlerContext ctx) throws Exception {
    super.channelReadComplete(ctx);
    try {
        ctx.channel().flush();
    } catch (Exception e) {
        e.printStackTrace();
    }
}

其中会调用DefaultChannelPipeline#flush，使用tail节点来冲刷缓冲区中所有未决的消息

public final ChannelPipeline flush() {
    tail.flush();
    return this;
}

看下冲刷缓冲区的时序图

这里又涉及到一次冲刷的事件传播，传播过程就不说了，这里主要看最后两步

  • 如果已经冲刷，或者缓冲区没有数据，则直接退出冲刷流程

  • 如果通道Channel处于非激活状态，则调用failFlushed将所有待处理的数据标记为失败，并且退出冲刷流程

  • 调用doWrite()执行冲刷流程，当出现异常时，直接调用close()就可使所有待冲刷的消息失败

    protected void flush0() {

  if (inFlush0) {
      return;
  }

  final ChannelOutboundBuffer outboundBuffer = this.outboundBuffer;
  if (outboundBuffer == null || outboundBuffer.isEmpty()) {
      return;
  }

  inFlush0 = true;

  if (!isActive()) {
      try {
          if (isOpen()) {
              outboundBuffer.failFlushed(new NotYetConnectedException(), true);
          } else {
              outboundBuffer.failFlushed(newClosedChannelException(initialCloseCause), false);
          }
      } finally {
          inFlush0 = false;
      }
      return;
  }

  try {
      doWrite(outboundBuffer);
  } catch (Throwable t) {
      if (t instanceof IOException && config().isAutoClose()) {
          initialCloseCause = t;
          close(voidPromise(), t, newClosedChannelException(t), false);
      } else {
          try {
              shutdownOutput(voidPromise(), t);
          } catch (Throwable t2) {
              initialCloseCause = t;
              close(voidPromise(), t2, newClosedChannelException(t), false);
          }
      }
  } finally {
      inFlush0 = false;
  }
}

总结

随处可见的EventLoop

在一次请求链路中，I/O线程和业务线程无缝地在做各种切换，Selector的轮询机制，会将感兴趣的事件封装成任务放置任务队列中执行。Netty中保证了，如果非I/O线程执行的操作，会将操作逻辑放置线程池中使用，从而保证了I/O线程的可利用性。

事件传播

在ChannelHandlerContext中带有fire***的方法，一旦被调用，意味着触发了一次事件传播。
