记录一次线程池引发的bug

背景

  • 系统中存在多数的调度任务执行

  • 项目使用的微服务架构是Spring Cloud，系统中服务间调用通过token来鉴权，即在header上添加token。于是就用上个了Threadlocal来管理token

    关于跨线程token授权，可以看下
    Spring Cloud 之 Hystrix 跨线程传递数据
    Hystrix传播ThreadLocal对象

最早的版本

原来的调度任务是通过@Schedule + @Async 来控制。
@Schedule线程池默认单线程，即在同一个项目中，一个任务在运行，另一个任务则需要排队等待
@Async实现了AsyncConfigurer，自定义了一套线程池策略，其中设置了拒绝策略是当任务添加到线程池被拒绝时，会在线程池当前运行的线程中执行被拒绝的任务。

@Component("taskExecutor")
@Slf4j
public class IThreadPoolTaskExecutor extends ThreadPoolTaskExecutor implements AsyncConfigurer {

    @Autowired
    private ConfigBean configBean;

    public IThreadPoolTaskExecutor() {
        // config something...
        // 设置拒绝策略
        this.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    }

    @Override
    public <T> Future<T> submit(Callable<T> task) {
        try {
            Future<T> future = super.submit(
                    new ICallable<T>(task, MDC.get(MdcConstant.REQUEST_ID_KEY), configBean.getToken()));
            return future;
        } finally {}
    }

}

注意，这里就有两个问题

  • @Schedule是单一线程池，系统中有多个调度任务执行时，不友好；
  • @Async的拒绝策略为线程池的所在线程执行拒绝任务，而@Schedule又是单一线程池，当系统中有多个调度任务在运行，某些情况就会出现@Async的线程池很容易就满了（因为一个调度任务中还有多个延时任务），导致线程池的所在线程在不断的执行任务。

而线上真实场景出现的问题就是，在某个时刻经常有些任务卡住，大概十几分钟，最长半个小时，在业务对于时效性比较敏感的前提下，这种问题是不允许存在的。

版本二

同事对其做了改进，直接通过@Schedule来控制，去掉@Async异步线程池的控制
@Schedule实现了任务调度接口，配置了线程池数量为10个

@Bean
public TaskScheduler taskScheduler() {
    ThreadPoolTaskScheduler scheduler = new ThreadPoolTaskScheduler();
    // 理论上并发线程最大数量为@Schedule的数量，目前@Schedule为8
    scheduler.setPoolSize(10);
    scheduler.setThreadFactory(new ThreadFactoryBuilder().setNameFormat("schedule-%d").build());
    return scheduler;
}

以上修改，放到生产，确实修复了第一版的拉取卡住问题。

然而在做后续的需求时，发现了更大的隐藏bug。。。

新需求

笔者主动尝试重构了某个模块，并在今天打算接入业务系统，刚好就要接入到调度任务中。
于是笔者按照版本二的思路，正常接入并放至测试环境，却出现问题了，经常出现服务间鉴权失败报401

而在同个调度任务中，其他未改动的service则正常跑。

排查思路

在开头也提到了，系统服务间调用是通过传递token来鉴权的，当未传递token时则会返回401的异常。代码原作者，还实现了一套自定义的并发策略来传递token，避免因为Hystrix的隔离策略导致获取不到线程副本中的token。详细可查看开头的链接。

笔者一开始整理的可能性如下

  • 代码里面涉及了多线程
  • 由于项目后续也接入了XXL-JOB来管理调度任务，初步怀疑XXL-JOB导致的问题
  • Hystrix隔离策略导致的问题，比如隔离策略为THREAD时，没办法拿到ThreadLocal的值（未真正验证过，网友所述）

根据以上可能性逐一排查

  • 代码中不存在多线程版本，排除
  • 将XXL-JOB去掉，继续使用@Schedule，还是会触发，排除
  • 隔离策略，由于对它了解的比较少，故参考了不少的文献后，决定排除。因为同样用一套代码，为什么任务中原有的逻辑都没问题，新改动的代码就有问题

再次梳理，其实这次的问题，肯定是出现在线程的问题上，排除了Hystrix线程的问题，故肯定是在其他线程的问题，现有线程池有一个@Schedule，还有一个@Async线程池，但是据同事所说已经去掉了@Async注解。

这时，突然在看原有逻辑的代码中，发现了@Async的注解，在调用的方法实现上添加的。

@Async
public void test() {
    // do something
}

而笔者的代码是按版本二的逻辑实现的，即只使用了@Schedule线程池，故大胆猜测@Async能够修复问题，确实如此！

有结果就可以反推问题，列出细节

  • 两个线程池
    • 调度线程池使用@Schedule（10个线程），或者接入XXL-JOB的话，只有一个线程
    • @Async 异步线程池（8个线程），拒绝策略是由调度线程执行任务
  • 调度线程池中维护着线程副本
  • 线程副本存放token，用于服务调用鉴权
  • 代码中清除线程副本涉及的位置很多

本地进行debug测试，在前几次拿到的线程副本如下，都不为空，并且可以观察下线程名

原来在threadlocalmap里是有副本信息的，再debug几轮就没了，如下图所示

基于以上，基本可以确定是因为线程策略的选择，导致线程副本被清空。

具体的清空地方如下，实现了Callable接口，并重写了call方法，
其中先获取主线程传递进来的线程副本token，再将token设置到当前线程的线程副本中（用于后续的调用，比如项目里是在RequestInterceptor的实现类中获取并设置header，再去进行Feign调用），当执行完任务后，当前线程会将线程副本的token清除掉。

@Slf4j
public class ICallable<T> implements Callable<T> {

    private Callable<T> task;
    private String requestId;
    private String token;

    public ICallable(Callable<T> task, String requestId, String token) {
        this.task = task;
        this.requestId = requestId;
        this.token = token;
    }

    @Override
    public T call() throws Exception {
        if (requestId != null) {
            MDC.put(MdcConstant.REQUEST_ID_KEY, requestId);
            FeignHolder.REQUEST_ID.set(requestId);
        }

        if (token != null) {
            FeignHolder.TOKEN.set(token);
        }

        try {
            return task.call();
        } finally {
            if (requestId != null) {
                MDC.remove(MdcConstant.REQUEST_ID_KEY);
                FeignHolder.REQUEST_ID.remove();
            }
            log.debug(FeignHolder.TOKEN.get());
            if (token != null) {
                FeignHolder.TOKEN.remove();
            }
        }
    }

}

举个栗子

当前使用调度线程池（10个），异步线程池（3个）（本地由于资源有限，故将异步线程池设置小点）

  1. 调度任务的第一步是先给当前调度线程设置线程副本，即设置token
  2. 顺利的话，异步线程池不紧张，项目正常进行，每个异步线程执行完任务后，接着执行ICallable的call方法，将当前线程的副本token清除
  3. 突然遇到异步线程池满了，调度线程接替执行，接着执行ICallable的call方法，在调度线程将token清除了
  4. 后续的任务，在执行ICallable#call时，无法再通过调度线程获取token，即后续链路中token 都为null，故服务调用就会报401

解决方案

原作者，将服务调用都通过这一套鉴权体系管理起来，链路大致如下

  • 用户调用接口时，若涉及服务调用，Hystrix会通过FutureTask提交任务，故在run的时候，就会来到ICallable#call，并将token传递到执行线程作为线程副本
  • 系统调用接口时，若涉及服务调用
    • 若使用异步线程池，则通过自定义异步线程池类，来submit任务，在run的时候，就会来到ICallable#call，并将token传递到执行线程作为线程副本
    • 若使用主线程（或调度线程），Hystrix会通过FutureTask提交任务，故在run的时候，就会来到ICallable#call，并将token传递到执行线程作为线程副本

如果使用的都是异步线程池的话，则不会有token被清除导致不能用的问题，因为它每次传入ICallable对象的token都是从配置读取的，但是调度线程的token终究还是被清除了。

方案一

同个调度任务中，服务调用的相关代码，均使用同一套异步线程池，保证服务调用不存在于调度线程。
缺点：治标不治本，调度线程的token仍然有机会被清除。当调度任务中添加新的服务调用代码，开发者无法感知到有这个bug，将其置于调度线程调用。

方案二

同个调度任务中，按需使用异步注解@Async，且更改异步线程池的拒绝策略，为非调度线程执行。
优点：可对任务中的子任务进行分类，较为延时的使用异步线程池。拒绝时则排队等待

方案三

同个调度任务中，涉及代码都不用异步注解@Async，可直接在最外层使用@Schedule线程池，自定义配置管理。

总结

此次排查问题比较久，主要问题在于

  • 项目日志没有加线程名标记，不清晰
  • 在未知答案的前提下，本地调试不容易发现问题
  • 对于线程池的策略选择不合理，当配置了拒绝策略为线程池所在线程执行时，极容易导致很多问题，比如主线程卡住，线程的副本”变脏“、”丢失“等等
  • 对于Spring cloud的了解不够深，导致将大部分时间花在了排除Hystrix隔离策略、Feign并发策略上
  • 版本二中，同事未发现该问题，以为只有一个@Scheudle线程池，也因此排除了这种可能性
